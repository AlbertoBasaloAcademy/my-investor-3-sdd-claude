### 1.2 `back/src/main/java/dev/aiddbot/abjavareact/booking/Booking.java`
- Add new field `passengerPhone`:
  ```java
  @Column(name = "passenger_phone", nullable = false)
  private String passengerPhone;
  ```
- Update constructor signature to take phone and drop client-settable status (status is always set internally to `CREATED` by the entity itself, not passed in):
  ```java
  public Booking(Launch launch, String passengerName, String passengerEmail, String passengerPhone) {
    this.launch = launch;
    this.passengerName = passengerName;
    this.passengerEmail = passengerEmail;
    this.passengerPhone = passengerPhone;
    this.status = BookingStatus.CREATED;
  }
  ```
  Rationale: matches the spec rule "a new booking always starts CREATED" at the entity level, not just in the service ÔÇö this is the strongest place to guarantee the invariant. JPA's no-arg `protected Booking()` constructor remains for Hibernate.
- Add `getPassengerPhone()` getter.
- Remove the public `setStatus(BookingStatus)` setter ÔÇö status now only changes via a dedicated `cancel()` behavior on the entity, never via arbitrary `setStatus`. Add a method that encodes the transition rule directly on the entity:
  ```java
  public void cancel() {
    this.status = BookingStatus.CANCELLED;
  }
  ```
  This keeps the "CREATEDÔåÆCANCELLED only, never back" rule colocated with the entity rather than re-implemented ad hoc in the service, and removes the only call site that could otherwise set status to an arbitrary value.

  Since `update()` is being removed entirely (per the already-decided design), `setLaunch`, `setPassengerName`, `setPassengerEmail` become dead code with no caller. **Recommendation: delete the unused setters** (`setLaunch`, `setPassengerName`, `setPassengerEmail`) to keep the entity surface minimal and matching the spec's lifecycle (only `cancel()` mutates state post-creation). Do not add a `setPassengerPhone` setter either ÔÇö nothing will call it. Final entity should expose only getters + `cancel()` plus the constructor.

## 2. Backend API changes

### 2.1 Final endpoint list after change
- `GET /api/bookings` ÔÇö list all (unchanged shape, new fields)

`validate()` ÔÇö update to check phone instead of status, and drop the status-null check:
```java

  if (request.launchId() == null) {
    throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "Launch is required");
  }

    throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "Passenger name is required");
  }
  if (request.passengerEmail() == null || request.passengerEmail().isBlank()) {
    throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "Passenger email is required");

    throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "Passenger phone is required");
  }

```

`cancel()` ÔÇö new method, loads the entity (it needs to mutate and save, not just check existence like `delete()` did):
```java
public BookingResponse cancel(Long id) {
  Booking booking =
      repository
          .findById(id)
          .orElseThrow(
              () -> new ResponseStatusException(HttpStatus.NOT_FOUND, "Booking not found: " + id));
  booking.cancel();
  return toResponse(repository.save(booking));
}
```
Note: spec doesn't require rejecting a *second* cancel of an already-CANCELLED booking (no acceptance criterion demands a 409/400 for double-cancel). `cancel()` on the entity is idempotent (sets status to CANCELLED regardless of current value) ÔÇö this doesn't violate "never transition back" since CANCELLEDÔåÆCANCELLED isn't a backward transition. The frontend gates the Cancel button on `status === 'CREATED'`, so this path is normally unreachable via the UI anyway. **Decision: do not add extra guard logic for double-cancel** ÔÇö keep it minimal per "no speculative abstractions." If stricter behavior is wanted later, that needs an explicit new acceptance criterion.

Remove `delete()` and `update()` entirely. `findLaunch()` private helper stays unchanged. `toResponse()` updated to include `passengerPhone`:
```java
private BookingResponse toResponse(Booking booking) {
  return new BookingResponse(
      booking.getId(),
      booking.getLaunch().getId(),
      booking.getLaunch().getRocket().getName(),
      booking.getLaunch().getDate(),
      booking.getPassengerName(),
      booking.getPassengerEmail(),
      booking.getPassengerPhone(),
      booking.getStatus().name());
}
```

### 5.2 Cross-component refresh problem (booking created in LaunchList, must show up in BookingList)

`LaunchList` and `BookingList` are independent siblings in `App.tsx` with no shared state. If a booking is created from `LaunchList`'s new inline form, `BookingList`'s own `useBookings()` has no way to know to refetch.

**Options considered**: (1) lift a refresh signal/callback through `App.tsx` props ÔÇö minimal, no new infra; (2) global event bus/context ÔÇö overkill for a 3-component app, rejected; (3) polling ÔÇö wasteful, rejected; (4) do nothing ÔÇö rejected, would visibly break the feature (new bookings wouldn't appear without a manual reload).

**Recommendation: option 1.** `App.tsx` holds a `bookingsVersion` counter, increments it via a callback passed to `LaunchList` (`onBookingCreated`), and passes the counter down to `BookingList` as `refreshSignal`; `BookingList` calls `refresh()` whenever `refreshSignal` changes (see 5.1 above for the `useEffect`). This is the one piece of "new" wiring beyond what's explicitly spelled out in the task ÔÇö flagged clearly because it's a necessary consequence of the already-decided "Book button moves into LaunchList" design choice, not a speculative addition.

  </Fragment>
))}

Test id naming: the new booking form uses **distinct** test ids from the existing launch edit/create form (`data-testid="booking-form-error"` and `data-testid="booking-submit-btn"`, NOT the existing form's `form-error`/`submit-btn`) because the two forms are independent UI surfaces that could in principle both be open simultaneously (user clicks Edit on one row and Book on another), which would make `screen.getByTestId('form-error')`/`getByTestId('submit-btn')` ambiguous in tests if shared. ## 7. Sequencing -- recommended implementation order

Backend first (compiles and is independently testable), then frontend, keeping the app buildable/testable at each checkpoint:

1. Backend data model: BookingStatus.java -> Booking.java (constructor + cancel() + remove dead setters) -> BookingRequest.java -> BookingResponse.java. The backend will NOT compile yet after this step alone (service/controller/tests still reference old shapes) -- expected, since steps 2-3 immediately follow before running tests.
2. Backend service + controller: BookingService.java (validate/create/cancel, remove update/delete) -> BookingController.java (add PATCH /cancel, remove PUT/DELETE). Main backend source now compiles cleanly.
3. Backend tests: BookingServiceTest.java -> BookingControllerTest.java -> BookingRepositoryTest.java. Run mvnw.cmd test -- should be green for the booking package; Rocket/Launch suites unaffected.
4. Frontend types + API layer: booking.ts -> httpClient.ts (add patch) -> bookingsApi.ts (remove update/delete, add cancel). BookingList.tsx/LaunchList.tsx will fail to typecheck at this point (still referencing removed functions/old types) -- expected transient state, immediately followed by step 5.
5. Frontend components: BookingList.tsx rewrite -> LaunchList.tsx (Book button, inline form, onBookingCreated prop) -> LaunchList.css (new .launch-booking-form rule) -> prune dead .booking-status--confirmed/--payed from BookingList.css -> App.tsx (wire bookingsVersion/refreshSignal/onBookingCreated). Run npm run build -- should compile cleanly.
6. Frontend tests: bookingsApi.test.ts -> useBookings.test.ts (fixture-only update) -> new BookingList.test.tsx -> LaunchList.test.tsx (add Book-flow tests + bookingsApi mock). Run npm run test.
7. Lint pass: npm run lint on frontend; fix any unused-import/unused-variable flags surfaced by the removals (e.g. leftover BookingStatus literal CONFIRMED/PAYED references, leftover Mockito then/willThrow/willDoNothing imports on the backend side per section 3).
8. Manual smoke test (see section 8).

Practical grouping: do steps 1+2+3 as one atomic unit before moving to frontend (do not try to run backend tests mid-step-1, since BookingServiceTest still references the old constructor until step 3 catches up). Likewise steps 4+5+6 as one atomic unit for frontend. Only the start (current main) and end (fully done) states, plus the boundary between backend-done/frontend-not-started, need to build cleanly -- that boundary is a safe point to pause/commit.
